⏺ Build succeeds. Here's a summary of the fixes:

  Root cause 1: UDP connection never fails (visca/visca.go:47)
  - net.Dial("udp", ...) just creates a local socket — it never contacts the
  camera. So reconnect always reported "Connected" even when the camera was
  unreachable.
  - Fix: After opening the UDP socket, send a VISCA IF_Clear command and wait up
   to 2 seconds for an ACK. If the camera doesn't respond, the connection fails
  with a clear error message.

  Root cause 2: Missing IP/port in error responses
  (handlers/handlers.go:248-255,282-290)
  - settingsResponse used omitempty on Label, IP, and Port. When connection
  failed, these were zero-valued and omitted from JSON, causing data.ip to be
  undefined in the JS client.
  - The error response didn't include the attempted IP/label/port at all.
  - The JS error handler hardcoded updateCameraList(data.cameras, '', 0, false)
  instead of using the response data.
  - Fix: Removed omitempty from Label/IP/Port, always populate them in error
  responses, and use data.ip/data.port in the JS error handler.

  Defensive fix: Trim whitespace on IP input (handlers/handlers.go:262-263)
  - Added strings.TrimSpace on label and IP to prevent accidental spaces from
  corrupting stored camera data.
